## BUG LIST (severity, fix)

**B1 ÔÇö CRITICAL (argmax risk). The softmax is computed over the WRONG tensor in the multi-token shape, AND the host softmax is fine but the weights gather uses `probs[i]` which is correct ÔÇö the real risk is the dtype of the gather.** Re-examining: the proposal's host softmax over 256 is correct. The genuine CRITICAL is **the tie-break in argsort does NOT match llama.cpp.** Confirmed: llama.cpp uses `std::sort` (CPU, `ops.cpp:8227`) with comparator `data[a] > data[b]` and **no index tiebreak** (`cmp_argsort`, L8184-8193) ÔÇö `std::sort` is **unstable**; the CUDA backend uses CUB `DeviceRadixSort`/`DeviceSegmentedSort` (`argsort.cu`), also not index-stable. The proposal's "stable, smaller index wins ties" is a **fabricated** rule matching neither backend. Impact: only matters on exact-equal softmax probs (rare with F32), so it will almost never flip argmax ÔÇö but it is not the "verbatim" match claimed. **Fix:** match the *reference you validate against*. Since you gate against llama.cpp CUDA, exact tie order is nondeterministic there too; use plain unstable descending sort and accept that exact ties are undefined in both. Do NOT advertise it as bit-exact tiebreak. Severity CRITICAL only as a claim; LOW as a practical argmax risk ÔÇö **demote to MEDIUM** and document.

**B2 ÔÇö HIGH (correctness). `w_scale` (`expert_weights_scale`) is dropped by assumption.** Proposal: "NO routed_scaling_factor (w_scale==0)". llama.cpp applies it when `!=0 && !=1` (L1641). **B3 ÔÇö HIGH (compile/logic). `expert_matvec` is a free `fn` on `impl HybridModel` but called as `Self::expert_matvec(e, ...)` from `moe_ffn` ÔÇö and it takes `e_: &Engine` while ignoring `self`.** It should be a free function or an `Engine` method; as an associated fn with no `self` it's callable but the signature `Self::expert_matvec(e, &moe.gate_exps, ...)` is fine.
