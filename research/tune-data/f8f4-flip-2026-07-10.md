## Anomalies

1. **Acceptance divergence** (A, B, D): prefill-only change should NOT alter draft acceptance. Observed deltas:
   - Model A: 68.1% → 63.9% (-4.2pp)
   - Model B: 74.0% → 65.7% (-8.3pp)
   - Model D: 68.2% → 76.2% (+8.0pp)
   - Model C: 68.3% → 69.4% (+1.1pp, within noise)

   **Hypothesis**: FP-order difference in prefill MMQ changes the KV cache numerics → draft scoring diverges. Self-consistency PASS gates only check argmax tokens, not logit distributions. The acceptance variance suggests the f8f4 tile introduces enough numerical perturbation to shift borderline draft accepts/rejects.

2. **E2E spec tok/s regression** (A, B): despite +3-4% pp-only wins, e2e spec shows -3.5% / -6.1% tok/s loss. Model D shows +7.2% e2e gain (anomaly direction reversed). Prime time consistently improves -4 to -5.6% across all models (expected from pp speedup).

   **Hypothesis**: The acceptance divergence dominates e2e throughput for A/B. Lower acceptance → more target passes → tok/s loss outweighs prime win. Model D's acceptance INCREASE (+8pp) flips the trade: faster prime + higher acceptance → +7.2% e2e win.

3. **Run-to-run acceptance variance**: Both runs within each arm are IDENTICAL (e.g., A baseline both 68.1%, A f8f4 both 63.9%). This is unusual — normally stochastic sampling introduces variance. Greedy decode (argmax) explains identical tokens but NOT identical acceptance counts unless the prompt/draft chain is deterministic. Confirms the divergence is systematic, not noise.

## Verdict

**Gates**: All models pass self-consistency (argmax tokens match).

**PP-only**: Clear win across all models (+3.9% to +6.3% pp1845, +3.1% to +5.8% pp6257). The e4m3 tile is faster than int8 for prefill.

**E2E spec**: Mixed / inconclusive:
- Models A, B: regression (-3.5%, -6.1%) due to acceptance collapse
- Model C: neutral (-0.3%, noise)
- Model D: win (+7.2%) due to acceptance improvement

**Prime time**: Consistent win (-4.0% to -5.6%) — confirms prefill speedup reaches e2e TTFT.

**Acceptance anomaly**: The FP-order change in the f8f4 tile alters KV cache numerics enough to shift draft acceptance by -8.3pp to +8.0pp. This is NOT a correctness bug (argmax tokens match), but a numerical stability issue that makes the flip decision load-bearing on acceptance direction.

---

## Recommendation

BW24_MMQ_F8F4=1 should NOT be the default flip until the acceptance divergence is understood. The pp-only win is real, but the e2e impact depends on whether acceptance improves or regresses — model D shows it CAN win (+7.2%), but models A/B lose. More investigation needed:

1. Profile the f8f4 tile's FP-order vs int8 (accumulation pattern, reduction order)
2. Check if switching to FP32 accumulators (vs FP16) in the f8f4 tile stabilizes acceptance
3. Run a larger N (e.g., 5 prompts × 2 runs) to see if acceptance direction is prompt-dependent

The flip is NOT broken (gates pass), but the acceptance variance makes it risky as a blind default.
