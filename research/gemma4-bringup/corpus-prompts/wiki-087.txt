--- title: token-bucket type: concept tags: [rate-limiting, traffic-shaping, algorithm, valkey, ai-orchestration] sources: [] updated: 2026-06-29 --- # token-bucket The token bucket is a rate-limiting and traffic-shaping algorithm built around two parameters: a bucket capacity `B` (the maximum number of tokens it can hold) and a refill rate `r` (tokens added per unit time, up to `B`).[^1] A request may proceed only if the bucket holds enough tokens; on success the cost is deducted, and if the bucket is empty the request is denied, delayed, or queued. Because tokens accumulate during idle periods up to the capacity, the algorithm permits short bursts of size `B` while still enforcing a long-term average rate of `r` — its defining advantage over a fixed-rate limiter, which would block legitimate spikes.[^1] This is the standard contrast with the leaky bucket: the leaky bucket smooths bursty input into a constant output stream, whereas the token bucket lets a saved-up burst through at full speed and only then falls back to the steady refill rate.[^1] In [[glide-mq]] the token bucket is the core of the AI-native rate
