--- title: vLLM Serving (the serving contract) type: concept tags: [serving, inference, continuous-batching, paged-attention, openai-api, tool-calling] sources: [] updated: 2026-06-29 --- # vLLM Serving (the serving contract) **vLLM serving** is the production-runtime behaviour of the [[vllm|vLLM]] engine when it is run as an HTTP service (`vllm serve <model>`) rather than as an offline batch library. The serving layer wraps the throughput core — **PagedAttention** (OS-style paged KV cache in fixed non-contiguous blocks, near-zero fragmentation, cross-request KV sharing) plus continuous (iteration-level) batching and chunked prefill — behind an **OpenAI-compatible HTTP API**, so existing client tooling drops in unchanged.[^1] The defining property of this serving model is *iteration-level scheduling*: requests join and leave the running batch every decode step instead of waiting for a fixed batch to complete, which is what lets a single server hold many concurrent sessions at high aggregate throughput.
