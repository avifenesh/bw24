--- title: Memory safety type: concept tags: [programming-languages, security, systems, rust] sources: [] updated: 2026-06-29 --- **Memory safety** is the property of a programming language (or program) that prevents a class of bugs arising from mishandling memory: out-of-bounds reads and writes, use-after-free, double-free, null-pointer dereference, uninitialized reads, and data races. Languages achieve it in different ways. Garbage-collected languages (Go, C#, Java, Swift, Python, JavaScript) manage memory automatically at runtime; Rust enforces it at compile time through its ownership and borrow-checking model with no garbage collector. C and C++ are *not* memory-safe by default — developers can reduce risk through disciplined coding, static analysis, and sanitizers, but the guarantee is not built into the language.[^1] A subtle catch: even code written in a "safe" language can lose its guarantees when it imports unsafe C/C++ libraries across a Foreign Function Interface (FFI), which is why projects like Omniglot (Princeton/UC Berkeley/UC San Diego) work on safe FFI boundaries.[^1] Memory-safety bugs have historically dominated serious vulnerabilities in large systems software.
