--- title: action-graph type: concept tags: [agent-tooling, agent-memory, provenance, graph] sources: [codex-mem-agent-tooling] updated: 2026-06-29 --- # action-graph An action-graph is the data structure at the heart of [[get-oriented]] (orientation): a file-level, transcript-derived record of what an agent actually did, so a later session can look up the real history of code it did not write rather than guessing. It is deliberately *not* a semantic code index — it maps goals to the files they touched and the outcome of that work, pointing an agent at the right area and the prior intent to read.[^1] The pipeline that builds it runs `ingest.js` (per-runtime adapters) → `raw.jsonl` → `condense.js` → `episodes.json` → `graph.js` → `graph.json` → `consume.js` → CLI, deriving all signal locally from transcript JSONL with no model call or external service.[^1] The graph holds two node kinds — `intent` nodes (user goals) and `file` nodes — joined by four edge kinds: `touched` (goal → file), `coupled` (files edited together), `outcome` (committed / failed / edited / explored), and `resumes` (the same work picked back up across detours).[^1] On top of
