--- title: Constrained Decoding type: concept tags: [inference, structured-output, json-schema, grammar, tool-calling, sampling, serving] sources: [] updated: 2026-06-29 --- # Constrained Decoding Constrained decoding (also called constrained sampling, grammar-constrained generation, or structured outputs) forces an LLM's token-by-token output to conform to a formal language — a JSON Schema, a regular expression, or a context-free grammar (CFG). The mechanism is a **token mask** applied at each decode step: given the grammar, the tokenizer, and the prefix generated so far, the engine computes the set of tokens that could still lead to a valid string and zeroes out (masks) the logits of every other token before sampling. This makes invalid output *structurally impossible* rather than merely discouraged by a prompt. It is the production answer to "make the model emit valid JSON every time," and it is what powers strict tool/function calling, where a tool call's arguments must match the tool's declared JSON Schema for the call to be machine-dispatchable.[^1] The headline cost is per-token CPU: a good engine computes a JSON-schema mask in roughly 50μs of single-core time for a 128k-vocab
