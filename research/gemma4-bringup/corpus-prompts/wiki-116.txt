--- title: Safe C++ type: concept tags: [cpp, memory-safety, borrow-checker, language-design, rust] sources: [] updated: 2026-06-29 --- **Safe C++** is a 2024 proposal (paper P3390R0, dated 2024-09-11, audience SG23) by Sean Baxter and Christian Mazakas to retrofit Rust-style memory safety onto C++ as a backward-compatible language superset.[^1][^2] Rather than hardening the existing language in place, it defines a new *safe* context (opted into per-function or per-file) in which only a rigorously safe subset of C++ is allowed: pointer arithmetic and other unsound operations are banned by the compiler frontend, lifetime errors are caught by static analysis, and the remaining cases such as out-of-bounds subscripts become runtime panics/aborts.[^2] The explicit goal is parity with the safety guarantees of Rust while keeping C++'s function overloading, templates, inheritance, and exceptions, so existing unsafe code keeps compiling and can be migrated incrementally.[^2] The reference implementation is Baxter's Circle compiler, which already implemented the borrow checker, relocation, and the new syntax.[^1] To make the safe subset *useful* once raw pointers and unions are removed, the proposal imports a large slice of the Rust model: a
