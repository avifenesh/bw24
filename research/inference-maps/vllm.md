# Implementation Map — vLLM (v1 engine, PagedAttention, FlashInfer, Marlin)

Reference engine: Python orchestration + CUDA/Triton/CUTLASS kernels, multi-request server with continuous batching, paged KV cache. Far from bw24's shape (bw24 is single-stream, C/CUDA, resident-quant GGUF), so most of vLLM's value is *ideas and layouts*, not kernels. This map catalogs techniques per subsystem with concrete `file:line` anchors and an sm_120 (Blackwell consumer / RTX 5090, RUNS/NEEDS-PORT/DEAD) fit verdict. All paths are under `/data/projects/bench-engines/vllm-venv/lib/python3.12/site-packages/vllm/`.

sm_120 reality check that governs every verdict below:
- HAS: `mma.sync` m16n8k8/k16/k32 (int8/fp16/bf16/fp8), `ldmatrix`, `dp4a`, FP8 E4M3/E5M2 math, TMA async-copy, `cudaStreamBeginCapture` (CUDA graphs), Triton-JIT to `mma.sync` tier.
- LACKS: `wgmma` (Hopper/sm_90 warpgroup MMA), `tcgen05` (sm_100 5th-gen tensor core), block-scaled MMA for NVFP4 (datacenter Blackwell sm_100 only), AMX (CPU, irrelevant).
- Consequence: every vLLM path that is "trtllm-gen" or NVFP4-KV or DeepGemm-FP8 or wgmma-FlashAttention is **DEAD or NEEDS-PORT** on sm_120; the *Marlin* GEMMs and the *Triton* attention kernels are the RUNS surface.

## What bw24 could take from vLLM

Highest-value, most-portable ideas for a single-stream sm_120 GGUF engine, in priority order:

1. **Marlin W4A16 GEMM (GPTQ/AWQ) — the proven sm_80+ `mma.sync` m16n8k16 weight-only kernel** (`marlin_utils.py:534-662`, scale permute `:26-29`; `_custom_ops.py:1376-1417`). 16-wide threadblock tile, per-group scales permuted into an 8×8 interleave, FP32 global reduce in workspace. No wgmma, no tcgen05 — runs natively on sm_120. This is the single most directly-portable quantized prefill GEMM in the whole engine and the canonical alternative to bw24's MMQ for 4-bit weights. Single-stream value: 24GB + batch=1 has the workspace headroom for the FP32 global reduce; the 16-tile is efficient even at small M.

2. **Marlin FP4/FP8 weight-only repack + scale-encoding layouts (no block-scale MMA needed)** (`marlin_utils_fp4.py:30-114,149-286`; `marlin_utils_fp8.py:42-199`). FP4 weights repacked to int32 Marlin columns, scales encoded as S0E5M3 (FP4) / fused-exponent-bias (FP8) — dequant is *memory-bound*, fed to the same m16n8k16 path. This is exactly how bw24 should treat MXFP4/NVFP4 GGUF on sm_120: dequant-on-load to the standard tensor-core tile, *not* the datacenter block-scale MMA. Min SM_75.

3. **2-stage Flash-Decoding split-KV Triton kernel (SGLang origin)** (`ops/triton_decode_attention.py:60-100`). Stage-1 computes per-split partial softmax over `NUM_KV_SPLITS` tiles, stage-2 merges. Lower register pressure than one-shot softmax, launch grid `(1, num_kv_heads, NUM_KV_SPLITS)` gives full occupancy at batch=1 by splitting along KV. This is the right *decode* attention shape for a single-stream engine and compiles straight to sm_120 `mma.sync`. Pairs with bw24's existing q8_0-K/q5_1-V cache.

4. **PagedAttention block table + hash-based prefix cache (pure CPU, zero kernel cost)** (`block_pool.py:130-183`, hash chain `kv_cache_utils.py:563-710`). Fixed-size KV blocks tracked by a per-request block-ID list; filled blocks hashed `SHA256(parent_hash || token_ids || extra)` into a `BlockHashToBlockMap` for prefix dedup. Even single-stream, a chat/agent loop re-sends the same system prompt every turn — prefix-cache hit skips recompute of that prefill entirely. All CPU-side; no GPU dependency. Highest ROI non-kernel idea.

5. **FP8 E4M3 KV cache with per-block scale + on-the-fly dequant-on-read** (`flashinfer.py:618-642,1401-1409`; layout `[num_blocks,2,block_size,num_kv_heads,head_size]` fp8). Device-agnostic FP8 math; `bmm1_scale = sm_scale*q_scale*k_scale`, `bmm2_scale = v_scale` fold scales into the matmul. 2x KV shrink, decode is bandwidth-bound so this is a direct batch=1 win. bw24 already does q8_0-K/q5_1-V; FP8-E4M3 is a cleaner, hardware-native alternative worth A/B-ing.

6. **CUDA-graph capture of a fixed batch=1 decode step** (`flashinfer.py:582-593`). Per-batch-size pre-planned wrapper, metadata captured once and replayed — kills per-op kernel-launch CPU latency. Single-stream decode is launch-bound (many tiny kernels per token), so a captured batch=1 graph is exactly the workload that benefits. Replay on sm_120 is fully supported.

7. **Online-softmax LSE-merge primitive (`merge_attn_states`) for cascade / split attention** (`flash_attn.py:1172-1223`). Running max/sum reweight to combine two independent attention passes. Architecture-neutral, the reusable core for any split-KV decode or shared-prefix split bw24 authors. (Take the *primitive*; skip the wgmma cascade kernel above it.)

8. **Greedy rejection-sampling kernel with zero-copy early exit** (`rejection_sampler.py:715-765`). For speculative/MTP decode: stage-1 argmax target logits, early-exit when draft == argmax, append bonus token — avoids a full-vocab softmax on the draft. bw24 already ships batched partial-accept MTP replay (commit abef37d); this is the matching accept/reject kernel and the early-exit pattern is a clean batch=1 fit.

## DEAD for bw24

Does not run on sm_120, or is pure multi-request server machinery with no single-stream value:

- **NVFP4 KV cache + trtllm-gen attention** — `flashinfer.py:98-210` (FP4 dequant Triton), `:618-642,1506-1614`, force-trtllm `:778,804`. NVFP4 KV uses trtllm-gen kernels gated to sm_100+ (B200/GB200); consumer Blackwell sm_120 lacks the block-scale MMA path. **DEAD** — fall back to FP8 KV cache (idea #5).
- **Block-scaled FP4/FP8 `scaled_mm` (tcgen05 swizzled)** — `_custom_ops.py:36-86` (`create_fp4_scale_tensor` swizzled layout), `compressed_tensors/triton_scaled_mm.py`, `BlockScaledMMLinearKernel.py`. `tcgen05` MMA + TMA scale-swizzle are sm_100-only. **NEEDS-PORT** — use channel/group-wise scales (`group_size=-1`) on the Marlin m16n8k16 path instead.
- **DeepGemm FP8 grouped GEMM for MoE** — `fused_moe/experts/deep_gemm_moe.py:49-100`. Assumes `wgmma` (sm_90+) for FP8 tensor cores; also self-disables for N<=512. **NEEDS-PORT/DEAD** — Triton FP8 MoE kernel is the portable substitute.
- **FlashInfer MLA decode (latent attention)** — `mla/flashinfer_mla.py:65-96,155-202`, explicit `capability.major == 10` (sm_100) gate. trtllm-gen exclusive, no prefill path. **NEEDS-PORT** — only worth a hand-ported Triton MLA kernel if a DeepSeek-class latent model is actually targeted.
- **FlashAttention cascade via `_vllm_fa3_C` (wgmma)** — `flash_attn.py:1054-1129,1172`. The prefix/suffix split kernel uses the FA3 `.abi3.so` (wgmma, sm_90). **NEEDS-PORT** — keep only the `merge_attn_states` LSE primitive (idea #7); the FA3 kernel itself is unusable on sm_120.
- **Marlin Int8 W8A16 MoE backend** — `fused_moe/experts/marlin_moe.py:59-100`. Codepath assumes the SM_80+ Marlin tile stream; per the findings the narrower sm_120 warp tile may mismatch, recommend Triton fallback for int8 MoE. **NEEDS-PORT** (the dense W4A16 Marlin RUNS; the MoE variant is unvetted on sm_120).
- **Continuous batching / preemption / priority eviction** — `sched/scheduler.py:376-560` (running queue), `:474-505,974-1000` (preempt). Multi-request only; batch=1 never preempts and never merges. **DEAD for bw24's single-stream purpose** (the scheduler runs, but adds only overhead).
- **KV transfer P/D + Context/Decode Parallel (DCP) all-gather** — `scheduler.py:116-136,2154-2188`; `flashinfer.py:1719-1743,213-322`. Cross-rank/cross-GPU KV movement + NCCL LSE reduction. Single-GPU single-stream gets no parallelism benefit, only collective-op latency. **DEAD**.
- **EP / EPLB expert parallelism + redundant-expert placement** — `model_loader/ep_weight_filter.py:31-82`, `qwen3_5.py:156-171`. Multi-rank expert sharding/load-balancing. Single GPU has one rank. **DEAD** (the EP weight *filter* I/O trick is irrelevant when all experts are local). N-gram CPU proposer (`ngram_proposer.py:169-286`) is CPU-only — useful only as a host draft source, not a GPU technique.
- **AMX** — not present in vLLM CUDA path; CPU-only, irrelevant.

## sm_120 summary verdict

- **RUNS / directly portable:** Marlin W4A16 + FP4/FP8 weight-only repacks (Subsys 3), 2-stage Flash-Decoding + Triton prefill (Subsys 2), PagedAttention block tables + hash prefix cache (Subsys 1), FP8 KV cache, CUDA-graph batch=1 replay, greedy rejection sampler + `merge_attn_states`.
- **NEEDS-PORT:** NVFP4 KV / trtllm-gen attention, block-scale `scaled_mm` (tcgen05), DeepGemm FP8 MoE, FlashInfer MLA, FA3 cascade kernel, Marlin Int8 MoE → all require dequant-to-mma.sync or Triton rewrites.
- **DEAD for bw24:** continuous batching / preemption / priority eviction, KV-transfer P/D, DCP/Context-Parallel all-gather, EP/EPLB expert sharding, AMX — all multi-request / multi-GPU machinery with no single-stream value.
