# SGLang — Implementation Map for bw24 (single-stream sm_120 GGUF engine)

Target: extract portable, high-value techniques for a **batch=1, single-stream, sm_120 (RTX 5090 / desktop Blackwell)** GGUF inference engine. SGLang is a *multi-request server* — most of its value for bw24 is in (a) the RadixAttention prefix-sharing trie, (b) Triton attention kernels that are explicitly sm_120-tuned, (c) speculative-decode (EAGLE/NEXTN) tree mechanics, and (d) the FP8/NVFP4 quant kernels that already detect `sm_120` and branch. The server scheduling machinery (TBO, disaggregation, multi-wrapper FlashInfer dispatch) is largely DEAD for batch=1.

`sm120_fit` legend:
- **RUNS** — compiles + runs on sm_120 today (Triton sm_120 codegen, or plain CUDA load/store, or pure CPU). Single-stream value noted inline.
- **NEEDS-PORT** — depends on an external compiled kernel (FlashInfer/CuTe-DSL/CUTLASS/Marlin) that lacks a native sm_120 path or auto-falls-back; the *algorithm* is portable but the kernel must be rewritten/retuned.
- **DEAD** — wgmma/tcgen05/TMA-persistent only, AMX-only, or multi-request-server-only with no batch=1 payoff.

## What bw24 could take from SGLang (top portable, highest-value)

1. **RadixAttention prefix trie (CPU-only, page-aligned).** A `TreeNode` radix tree keyed on token IDs, with `split_node()` to materialize mid-segment boundaries, `match_prefix()` returning the longest cached prefix as a device-index tensor, and `lock_ref` counters. For bw24 this is the single biggest win: a multi-turn chat or repeated-system-prompt session reuses the KV of the shared prefix for *free*, and it is pure CPU code with zero kernel dependency. `…/sglang/srt/mem_cache/radix_cache.py:206-267` (match), split logic earlier in same file. **RUNS.**

2. **Triton two-stage decode attention with dynamic KV-splits.** Stage-1 computes partial softmax + LSE per KV-split; stage-2 merges. For batch=1 the split count collapses to 1, giving clean linear work and avoiding any wgmma dependency. This is a drop-in replacement strategy for a hand-written GGUF decode-attention kernel and is *already* the sm_120 path in SGLang. `…/sglang/srt/layers/attention/triton_ops/decode_attention.py` + `triton_backend.py`. **RUNS.**

3. **sm_120-aware Triton block tuning.** SGLang detects `CUDA_CAPABILITY[0]==12` and selects smaller attention tiles (e.g. 64×128 for Lq≤128) to fit the ~100 KB shared-mem budget, and sets `num_stages=1` for FP8 quant kernels to dodge occupancy pressure on desktop Blackwell. bw24 should copy these exact heuristics rather than rediscover them. Attention: `triton_backend.py` (cap-12 branch); FP8: `…/sglang/srt/layers/quantization/fp8_kernel.py:1393` (`num_stages = 1 if _is_sm120_supported else …`). **RUNS.**

4. **HiCache GPU↔CPU KV tiering via plain vector-copy CUDA kernel.** `HiCacheKernel<kElementSize,kUnroll,…>` moves 128 B granules with `L1::no_allocate` (non-temporal) loads, no tensor cores. For a single-stream engine this is exactly the right primitive to spill cold prefix KV to host RAM (and back) at ~150–200 GB/s — letting bw24 keep a much larger effective KV history than VRAM alone allows. `…/sglang/jit_kernel/csrc/hicache.cuh:80-116`. **RUNS.**

5. **KV-cache quantization (FP8 K, FP8/FP16 V) with per-token/block scale, fused on store.** Quant happens inside `set_kv_buffer` so there is no separate pass; the Triton dequant path targets sm70+ and runs on sm_120. This is the cheapest 2× KV-memory shrink available and composes with the radix trie. `…/sglang/srt/mem_cache/common.py:27-91` (set_mla_kv_buffer / store paths). **RUNS.** (bw24 already ships q8_0 K / q5_1 V — this confirms the fused-on-store pattern is the right shape.)

6. **EAGLE/NEXTN speculative tree + greedy verify, all on portable kernels.** Draft worker emits topk tree, `build_tree_kernel_efficient` builds the attention mask (dense or bitpacked) via Triton, and `verify_tree_greedy` walks the tree with `sgl_kernel` — no wgmma. The **frozen-KV MTP (NEXTN)** variant is especially attractive for bw24: the draft reuses the target's frozen KV at one position (no draft KV pool, no allocation), which is the simplest possible spec-decode to bolt onto a single-stream engine. Tree: `…/sglang/srt/speculative/eagle_utils.py:48-159`; frozen-KV: `…/sglang/srt/speculative/frozen_kv_mtp_worker.py:81-199`. **RUNS.**

7. **EAGLE draft CUDA-graph capture of the whole N-step loop.** The N-step draft is captured into one CUDA graph and replayed, killing per-step launch overhead — directly applicable to bw24's batch=1 decode where launch latency dominates. `…/sglang/srt/speculative/eagle_draft_cuda_graph_runner.py:61-249`. **RUNS.**

8. **NVFP4 block-scaled GEMM with an explicit sm_120 CUTLASS path.** `nvfp4_scaled_mm_sm120.cuh` uses `MMA.SYNC m16n8k16/k32` with `nv_float4_t<e2m1_t>` block-scale — no wgmma/tcgen05. If bw24 ever wants FP4 weight GEMM on the 5090, this is the reference layout (128×128×128 tile, FP4 2-per-uint8, fp32 accum). `…/sglang/jit_kernel/csrc/gemm/nvfp4/nvfp4_scaled_mm_sm120.cuh:20-100`. **RUNS.**

## DEAD for bw24 (do not port)

- **MXFP4 persistent/TMA cutlass MOE path** — `mxfp4.py:~211` literally comments "SM120 desktop Blackwell does not support persistent/TMA MXFP4 path"; only a hand-patched non-persistent Triton fallback exists, with poor batch=1 occupancy. `…/sglang/srt/layers/quantization/mxfp4.py:160-252`.
- **Marlin INT4/FP4 GPTQ GEMM** — `marlin_utils.py:~90` returns empty support list for `device_capability < 80` and has *no* sm_120 native variant (needs persistent warps / TMA it doesn't have). `…/sglang/srt/layers/quantization/marlin_utils.py:53-149`.
- **Two-Batch Overlap (TBO) scheduling** — splits a batch into two halves to overlap compute/comm across two CUDA streams. Requires batch≥2; **disabled at batch=1**. `…/sglang/srt/batch_overlap/two_batch_overlap.py:59-150`.
- **CuTe-DSL GDN decode kernel** (`cutedsl_fused_sigmoid_gating_delta_rule_update`) — compiled C++/CuTe extension; use the Triton GDN fallback instead. `…/sglang/srt/jit_kernel/cutedsl_gdn.py`.
- **Multi-wrapper FlashInfer dispatch + W4A8 CUTLASS MOE** — FlashInfer's sm_120 support is version-gated and cutlass paths are SM_90/SM_100-tuned (NaN on heterogeneous batches → auto-fallback to cudnn). The multi-request paged-wrapper machinery has no batch=1 payoff. `…/sglang/srt/layers/attention/flashinfer_backend.py:114-890`; `…/sglang/srt/layers/quantization/w4afp8.py`.
- **Disaggregation / KV-sender / adaptive multi-graph server state** — `ScheduleBatchDisaggregationDecodeMixin`, prefill/decode-split transport: pure multi-node server machinery. `…/sglang/srt/managers/schedule_batch.py`.
- **Hybrid linear-attention dispatcher as a *server feature*** — GDN/Mamba layers themselves can run (Triton), but the model-zoo dispatcher (`HybridAttnBackend`) is only relevant if bw24 serves Qwen3.5-style hybrid models; the full↔linear router itself has no kernels.

### bw24 integration notes
- **Prefill gap (43× vs llama.cpp, per recent commit):** the **Triton unified-extend kernel** (`extend_attention.py:35-200`) with the cap-12 block tuning (64×128 for Lq≤128) is the most directly relevant reference — it is the deterministic single-pass prefill path SGLang itself runs on Blackwell desktop.
- **Decode latency:** combine the Triton 2-stage decode kernel (split→1 at batch=1) + whole-loop CUDA-graph capture (`eagle_draft_cuda_graph_runner.py` pattern generalizes to non-spec decode).
- **KV memory:** the **fused-on-store FP8 KV quant** (`common.py:27-91`) + **HiCache non-temporal copy kernel** (`hicache.cuh:80-116`) together give 2× shrink plus host spill — both pure-CUDA / Triton, both RUNS.
- **Easiest spec-decode:** **frozen-KV MTP / NEXTN** (`frozen_kv_mtp_worker.py:81-199`) — no draft KV pool, reuses target KV at one position; cheapest to bolt onto a single-stream engine.
- **Avoid:** Marlin and MXFP4-persistent paths on sm_120; all TBO / disaggregation server machinery.
