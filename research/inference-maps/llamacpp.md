# Implementation Map — llama.cpp (ggml-cuda)

Closest analog to bw24: C/C++ + raw CUDA, GGUF resident-quant weights, single-stream-capable. This map catalogs the techniques per subsystem with concrete `file:line` anchors and an sm_120 (Blackwell consumer / RTX 50-series, RUNS/NEEDS-PORT/DEAD) fit verdict. All paths are relative to `/home/avifenesh/projects/llama.cpp`.

sm_120 reality check that governs every verdict below:
- HAS: `mma.sync` m16n8k8/k16/k32 (int8/fp16/bf16), `ldmatrix`, `dp4a`, `__shfl_*_sync`, `cudaStreamBeginCapture`, fast `expf`.
- LACKS: `wgmma` (Hopper/Blackwell-datacenter `sm_90`-class warpgroup MMA), `tcgen05`, block-scaled `mma.sync` m16n8k64 for MXFP4/NVFP4 (`BLACKWELL_MMA_AVAILABLE`-gated), AMX (CPU, irrelevant).

## What bw24 could take from llama.cpp

The highest-value, most-portable ideas for a single-stream sm_120 GGUF engine, in priority order:

1. **MMQ register-dequant-to-int8 + `mma.sync` m16n8k32 GEMM** (`ggml/src/ggml-cuda/mmq.cuh:317-2930`). Quantized GGUF bytes are unpacked to packed int8 in registers during `load_tiles_q*`, then fed to integer tensor-core `mma.sync`. This is the canonical resident-quant GEMM and runs natively on sm_120 — no wgmma, no datacenter-only path. Single-stream value: this is the prefill matmul; it is the single largest lever for closing the documented prefill gap vs llama.cpp.

2. **MMVQ ↔ MMQ batch-aware dispatch** (`ggml/src/ggml-cuda/mmvq.cu:109-277`, `mmq.cu` dispatcher). Pure host-side decision: batch ≤ `MMVQ_MAX_BATCH_SIZE` (8) with short seq → MMVQ warp-per-row dp4a (latency-optimal for decode batch=1); else MMQ tiled GEMM (throughput-optimal for prefill). bw24 is exactly the single-stream case this split was tuned for: decode goes MMVQ, prefill goes MMQ.

3. **Inline-dequant Flash-Attention vec (decode) kernel** (`ggml/src/ggml-cuda/fattn-vec.cuh:86-254`, dequant in `fattn-common.cuh:408-617`). Warp-per-token, q8_0-K / q5_1-V dequantized on-load via bitwise ops + dp4a, online-softmax in registers. No tensor cores at all on the decode path. bw24 already ships q8_0-K/q5_1-V cache (commit 9ebf958); this is the matching attention kernel shape and maps 1:1 to batch=1 decode.

4. **Online-softmax tiling primitive** (`ggml/src/ggml-cuda/fattn-wmma-f16.cu:227-280`). Running `KQ_max`/`KQ_sum` with `exp(old_max - new_max)` rescale and the `FATTN_KQ_MAX_OFFSET = 3·ln2` numeric-range shift. Architecture-neutral; only needs `warp_reduce_max/sum`. This is the reusable core for any bw24-authored attention kernel (and what its hand-written sm_120 prefill kernel must reproduce).

5. **CUDA-graph capture-and-replay with 2-call warmup** (`ggml/src/ggml-cuda/ggml-cuda.cu:4468-4525`). `cudaStreamCaptureModeRelaxed` capture after two identical-property calls, replayed thereafter to kill per-op CPU launch latency. Single-stream decode (many tiny kernels, launch-bound) is the exact workload that benefits most (~3-5% typical). Property-stability check is CPU-side and cheap.

6. **Quantized K-shift via dequant→Hadamard→RoPE→requant graph** (`src/llama-kv-cache.cpp:1826-1876`, Hadamard table at `:336-369`). Lets RoPE position-shift operate on a quantized K cache without precision collapse, using pre-computed Walsh-Hadamard rotate matrices reused across layers. Directly relevant since bw24 holds a quantized K cache; the rotate-back/rotate-fwd trick is the non-obvious correctness piece.

7. **On-the-fly Q8_1 activation quantization pre-pass** (`mmq.cuh:28-59` structs, `mmq.cu:136-143` dispatch). A separate vector kernel quantizes F32 activations to the `block_q8_1_mmq` layout (interleaved scales + clipped int8) before the GEMM, so the matmul thread does zero per-element activation dequant. Cheap, arch-neutral, and a prerequisite for the int8 `mma.sync` path in idea #1.

8. **Per-dtype template-instantiation build structure** (`ggml/src/ggml-cuda/template-instances/mmq-instance-*.cu`, selector `mmq.cuh:4056-4141`). One kernel per GGUF quant type via `DECL_MMQ_CASE`, with per-dtype tile-size macros. Keeps binary size and per-dtype tuning sane; a clean pattern for bw24 to mirror as it adds quant types.

## DEAD for bw24

These do not run on sm_120, or are pure multi-request server machinery with no single-stream value:

- **Block-scaled MXFP4/NVFP4 `mma.sync` m16n8k64** — `mmq.cuh:945` (`#ifdef BLACKWELL_MMA_AVAILABLE`), `mma_block_scaled_fp4` call at `mmq.cuh:1056`, guarded regions `mmq.cuh:148-152, 247-251`, `vec_dot_fp4_fp4_mma` at `mmq.cuh:996-1065`. `block_scale` MMA is datacenter-Blackwell only; **NEEDS-PORT/DEAD on sm_120** — must fall back to dequant-FP4→int8 then the standard Q8_0 `mma.sync` m16n8k32 path. The FP4 *load_tiles* (`mmq.cuh:836-989`) layout is reusable; the MMA call is not.
- **WMMA tile prefill via `nvcuda::wmma`** — `fattn-wmma-f16.cu:15` (`namespace wmma = nvcuda::wmma`), fragment kernel `:75-222`. Volta-WMMA-intrinsic path; **NEEDS-PORT** — sm_120 must use explicit `mma.sync` tiles (`fattn-mma-f16.cuh`) instead, which lack a tuned sm_120 config. Not "DEAD" but unusable as-is.
- **Stream-K multi-block fixup scheduler** — `fattn-common.cuh:719-909`. An occupancy optimization for high-parallelism / many-request workloads; single-stream batch=1 has too little parallelism to need it. RUNS but **DEAD for bw24's purpose**.
- **Multi-stream KV cache (`n_stream = n_seq_max`)** — `llama-kv-cache.cpp:80-169, 250-257, 826-864`. Per-sequence ring buffers for concurrent requests. bw24 is unified single-stream (`n_stream = 1`); the N-way machinery is **DEAD**.
- **Dual-cache DSA indexer & independent-SWA cache wrappers** — `llama-kv-cache-dsa.cpp:14-172`, `llama-kv-cache-iswa.cpp:14-152`. Model-architecture-specific (DeepSeek DSA / sliding-window) dual-cache plumbing; not core single-stream path. Run only if those exact models are targeted; otherwise **DEAD**.
- **AMX** — not present in ggml-cuda; CPU-only, irrelevant to an sm_120 engine.
- **CPU MoE expert offload (`--n-cpu-moe`)** — `common/fit.cpp:527-549`, `common/arg.cpp:2329-2336`. Useful only for fitting oversized MoE models in limited VRAM; orthogonal to GPU single-stream perf. Keep on the shelf, not a core technique.

### Cross-cutting notes for the bw24 port

- The **MMQ int8 mma.sync path** (Subsystem 1) and the **FA vec decode kernel** (Subsystem 2) are the two pieces that map cleanest onto sm_120 with zero algorithmic redesign; both are already the documented gap (`bench: prefill pp` commit `8d1c0b7`) and the documented strength (`q8_0 K / q5_1 V` commit `9ebf958`) of bw24, so they are the lowest-risk, highest-return ports.
- Every **tensor-core prefill attention** path is `nvcuda::wmma` or untuned `mma.sync` configs (`fattn-wmma-f16.cu:75-222`, `fattn-mma-f16.cuh:10-87`) — a hand-written sm_120 `mma.sync` m16n8k{16,32} tile kernel reusing the online-softmax primitive (`fattn-wmma-f16.cu:227-280`) is the required net-new code.
- The only true dead ends are the **`BLACKWELL_MMA_AVAILABLE` block-scaled FP4 MMA** (`mmq.cuh:945-1065`) and the **multi-request server machinery** (multi-stream cache, stream-K, DSA/SWA dual caches) — none of which a single-stream engine needs.
