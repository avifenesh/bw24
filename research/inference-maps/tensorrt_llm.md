# TensorRT-LLM Implementation Map (via nv_internal kernels bundled in FlashInfer)

Scope: TensorRT-LLM kernels as vendored inside FlashInfer's wheel. Two trees:

- **XQA decode attention** — `flashinfer/data/csrc/xqa/` (the "XQA" multi-query/MLA decode kernels NVIDIA ships under TRT-LLM).
- **TRT-LLM CUTLASS / deep_gemm quant-GEMM** — `flashinfer/data/csrc/nv_internal/tensorrt_llm/{kernels/cutlass_kernels,deep_gemm}/`.

All paths below are rooted at:
`/home/avifenesh/.venvs/torch/lib/python3.12/site-packages/flashinfer/data/csrc/`

Target for bw24: **single-stream, batch=1, sm_120 (Blackwell consumer / RTX 50xx), GGUF weights.** `sm120_fit` column verdicts: RUNS = compiles + runs single-stream on sm_120 as-is or with trivial guard relax; NEEDS-PORT = mechanism is portable but the shipped instantiation/dispatch/guard excludes sm_120; DEAD = depends on hardware sm_120 does not have (wgmma/tcgen05) or on multi-request server machinery irrelevant to batch=1.

Verified architectural fact that corrects a finding: in `fpA_intB_gemm_template.h:433` the dispatch is
`} else if ((sm_ >= 80 && sm_ < 89) || sm_ >= 100) {` → **sm_120 (>=100) is routed into the `cutlass::arch::Sm80` mma.sync path**, NOT to an "unsupported" throw. So fpA_intB weight-only GEMM already has a live sm_120 compile path through the Sm80 tile configs; the porting work is tuning tiles, not adding a branch.

## What bw24 could take from tensorrt_llm

Ranked by portability × value for a single-stream sm_120 GGUF engine:

1. **MLA/MHA decode via `mla_sm120.cu` (TMA paged-KV + block-scale FP8, mma.sync m16n8k32).** This is the single most valuable artifact: a decode attention kernel *already authored for sm_120*, using TMA for paged KV loads, block-scale FP8 K/V, `mma.sync.m16n8k32` (no wgmma), warp specialization with 8 math warps, `tokensPerTile=64` for FP8 / `32` for BF16. It is the template for bw24's decode path. `xqa/mla_sm120.cu:40-53,77-147`.
2. **Inline dequant-in-the-MMA-feed pattern (`convertKCacheWordToF16`).** KV stored quantized in shared memory, unpacked to the math type *immediately before* the MMA reads it, no separate dequant pass. Directly applicable to bw24's q8_0 K / q5_1 V cache — same shape as what task #2 already shipped, but XQA's version fuses the unpack into the K-buffer staging loop. `xqa/mha.cu:1035-1037,1082,1267-1268`.
3. **Online softmax with split-KV (partition-KV) merge.** Pure-arithmetic per-warp running max/sum + a partial-output merge kernel using barriers/atomics. Hardware-independent, runs on sm_120 verbatim. Lets a long context be split across CTAs and reduced — useful when batch=1 underfills the GPU. `xqa/mha.cu:161-300`, `xqa/decode.cuh:62-116`.
4. **Two-buffer K + two-buffer V latency hiding for batch=1.** `nbKBuffers=2`, `nbVBuffers=2`, double-buffered async loads overlapped with MMA via CtaBarrier/mbarrier. This is exactly the occupancy/latency lever bw24 task #5 needs (decode is bandwidth/latency bound at batch=1). `xqa/mha.cu:76-99` (CTA/warp shape), `xqa/tma.h:76-128` (async load+barrier).
5. **fpA_intB weight-only quant GEMM (FP16/BF16 act × INT4/INT8 weight, group-64/128 or per-column scales) dequantizing inside the warp MMA.** Closest analogue to a GGUF quant-GEMM: weights stay INT, activations stay FP, dequant happens per-group at MMA-feed time. Already has an sm_120 compile path (Sm80 branch). This is the structural answer to bw24 task #10 (tensor-core quant GEMM). `cutlass_kernels/fpA_intB_gemm/fpA_intB_gemm_template.h:433` (sm_120→Sm80), `:517-527` (per-column QuantOp).
6. **FP8 amax→scale→quantize helper kernels (`scale=448/amax`, 1×128 and 128×128 granularity).** Plain load/reduce/store, no tensor cores; the building block for producing FP8 activations/weights for any FP8 path. Relax the `__CUDA_ARCH__>=890` guard and they run on sm_120. `deep_gemm/fp8_gemm_impl.cuh` (scale_1x128 / scale_128x128 region).
7. **ColumnMajorTileInterleave B-weight layout.** The interleaving that makes INT4/INT8 weights cheap to load+dequant in the MMA pipeline. Layout is hardware-independent; bw24 can adopt it for GGUF block layouts to keep the dequant feed contiguous. `cutlass_extensions/include/cutlass_extensions/gemm/kernel/mixed_gemm_B_layout.h`.

## DEAD for bw24

- **`mha_sm90.cu` + `gmma.cuh`/`gmma_impl.cuh` (wgmma.async path).** `wgmma.async`, `MatDesc`/`SwizzleMode`, `wgmma.fence/commit_group/wait_group` are Hopper-only (SM90+). The instruction does not exist on sm_120; this `.cu` will not compile/run there. Use `mha.cu` (generic mma.sync) or `mla_sm120.cu` instead. `xqa/gmma.cuh:28-50`, `xqa/mha_sm90.cu`.
- **deep_gemm Hopper cooperative FP8 GEMM (`cooperative_1x128_by_128x128_fp8_gemm_kernel`).** Guarded by `#if (__CUDA_ARCH__ >= 900)` at `deep_gemm/fp8_gemm_impl.cuh:452`; the math warp-groups use `cutlass::arch::warpgroup_reg_alloc` + wgmma. DEAD on sm_120 as written. (The deep_gemm *JIT* shell could in principle retarget, but only by emitting mma.sync — that is a rewrite, listed as NEEDS-PORT below, not a take.)
- **In-flight / continuous batching, paged-KV page allocator, multi-request scheduler, beam-search batching.** `batch_decode.cu` page-list management, semaphore-coordinated multi-request reductions, beam width plumbing — all multi-request server machinery. bw24 is single-stream batch=1; the page-table abstraction is overkill (a single contiguous KV buffer suffices). The *split-KV* reduction (item #3 above) is the one piece worth salvaging from this complex.
- **AMX / CPU-side paths.** Not present in these CUDA trees; no action.
- **FP4 KV-cache path in generic `mha.cu`.** `ENABLE_4BIT_KV_CACHE` (CACHE_ELEM_ENUM==3) carries a "not implemented" assert in the generic kernel and is absent from `mla_sm120.cu`'s is_fp8/is_bf16 branches — no working sm_120 instantiation exists, so it is DEAD-as-shipped (revisitable only as new-kernel work).

## Notes for the bw24 build decision

- The decode story for sm_120 is essentially **`mla_sm120.cu` + `mma.cuh`** (mma.sync m16n8k16/k32) + the **inline dequant** + **double-buffer/TMA** + **online-softmax/split-KV** pieces. None of that needs wgmma. This is buildable on sm_120 today.
- The quant-GEMM story splits cleanly: **fpA_intB already compiles for sm_120 via the Sm80 branch** (tune, don't port); the **Ada SM89 FP8 block-scale kernel** is a guard-relax + recompile (same mma.sync m16n8k32 sm_120 has); the **Hopper wgmma / deep_gemm-as-shipped is DEAD** and only reachable by rewriting the JIT template to emit mma.sync.
- Everything tied to **multi-request batching / paging / beam search** (`batch_decode.cu` page allocator, semaphore multi-request reductions) is **DEAD for a single-stream batch=1 engine** — extract only the split-KV reduction.
