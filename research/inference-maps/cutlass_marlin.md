# Implementation Map: cutlass_marlin (CUTLASS + Marlin quant-GEMM)

Scope: the two quant-GEMM kernel families most relevant to bw24 prefill on sm_120 (Ada GeForce, single-stream, GGUF weights):

1. **Marlin** (W4A16 / FP4 GEMM in vLLM/SGLang `sgl-kernel`) — hand-written CUDA, `mma.sync m16n8k16`, runs unmodified on sm_80+ including sm_120.
2. **CUTLASS FP4/FP8 block-scaled** (`nvidia_cutlass_dsl` + `flashinfer` `dense_blockscaled_gemm_sm120_b12x`) — CuTe DSL, native sm_120a/sm_120f block-scaled `mma.sync m16n8k64`, TMA + TMEM scale factors.

`sm120_fit` legend: **RUNS** = compiles+runs on sm_120 with no instruction swap; **NEEDS-PORT** = depends on something not present on sm_120 (sm_90 wgmma / sm_100 tcgen05 / TMA on data we don't have) and must be reworked; **DEAD** = architecturally irrelevant to a single-stream sm_120 GGUF engine. "single-stream value" = worth for batch=1 / one in-flight GEMM.

## What bw24 could take from cutlass_marlin

Ranked by portability x value for a single-stream sm_120 GGUF prefill engine. All file:line refs are in the FINDINGS source trees.

1. **Marlin's `mma.sync.aligned.m16n8k16.row.col.f32.f16.f16.f32` core** — `marlin_template.h:74-99`. This is the whole ballgame: a warp-level tensor-core MMA that RUNS verbatim on sm_120 (it's an sm_80 instruction). bw24's prefill GEMM is currently the 43x gap vs llama.cpp (commit 8d1c0b7); this is the smallest, most-proven path to tensor-core throughput. FragA=4 regs, FragB=2 regs, FragC=4 f32 accumulators. No wgmma, no TMA, no tcgen05. **Take this first.**

2. **Dequant-in-shared via `lop3` + magic constants** — `dequant.h:102-158` (INT4), `:358-388` (FP4 E2M1). LOP3.B32 LUT extraction of 4-bit weights from int32 packing directly into half2 fragments, fused into the load pipeline. GGUF Q4_K/Q4_0 blocks map cleanly onto this: unpack nibble -> half2, multiply by block scale. Runs on sm_80+. This is the GGUF-quant <-> tensor-core bridge bw24 needs.

3. **4-stage `cp.async.cg` + `ldmatrix` software pipeline with XOR-swizzled shared memory** — `marlin.cuh:47-79` (asm wrappers), `marlin_template.h:612-628` (XOR permute), `:818-829` (fetch_to_registers). cp.async.cg (16B cache-global) and ldmatrix.sync.aligned.m8n8.x4 are sm_80+ and bank-conflict-free via `row^(row%8)` XOR addressing. This is the latency-hiding skeleton; it is architecture-agnostic tuning, not blocked by hardware. Directly reusable to feed the m16n8k16 core.

4. **Marlin scale-permutation + repack offline transform** — `marlin_utils.py:302-323` (get_scale_perms / marlin_permute_scales), `gptq_marlin_repack.cuh:28-100+` (repack kernel). Pure index/arithmetic layout of weights+scales so the in-kernel fetch is contiguous and matches MMA fragment order. bw24 can run this once at model-load time (GGUF -> Marlin layout) so the hot loop reads aligned 16-wide K-blocks. No hardware dependency.

5. **Block/global scale application after MMA** — `marlin_template.h:335-338` (load global_scale), `:186-192` (sub_zp / scale_and_sub), `dequant.h:375-388` (FP4 E2M1 bias correction `BIAS_OFFSET=16`). The per-channel scale + optional zero-point applied as `__hmul2`/`__hfma2`/`__hsub2` on the FP32->FP16 path. GGUF per-block scales (and Q4_K's per-sub-block scale/min) slot directly into this fused-multiply step. Runs on sm_120.

6. **`atomicAdd` / `red.relaxed.gpu.global.add` split-K reduction with FP32 accumulate** — `marlin_template.h:225-265`, `:1180-1210`. For batch=1 with a thin M, Marlin reduces partial K-tiles across SMs via spinning barrier + relaxed atomics, with an optional FP32 reduce buffer for accuracy. bw24's single-stream prefill can use the FP32-reduce path to avoid FP16 accumulation drift on long K. Standard sm_80+ atomics.

7. **(Optional, medium value) NVFP4/FP4 dequant path for future FP4 GGUF** — `dequant.h:358-388`. If bw24 ever ingests NVFP4/E2M1 weights, Marlin's FP4 path (mask `0x70007000`, E2M1 bias) runs on sm_120 *without* needing the native block-scaled MMA. This is the "FP4 today, no sm_120a-only kernel required" escape hatch.

## DEAD for bw24

Architecturally irrelevant to a single-stream sm_120 GGUF engine. Do not port.

- **CUTLASS native block-scaled `mma.sync m16n8k64` MXF4NVF4 path** (`nvidia_cutlass_dsl/.../warp/mma.py:392-498`; `flashinfer/.../dense_blockscaled_gemm_sm120_b12x.py`). It *does* target sm_120a/sm_120f and RUNS there, but: (a) it requires **NVFP4/MXFP4 block-scaled tensors with E4M3/E8M0 scale factors in a specific TMEM layout** — bw24 has GGUF K-quant blocks, not NVFP4; (b) it carries the full **CuTe DSL / TMA / TMEM machinery** (288-thread 9-warp CTA, TMA load+store, scale factors preloaded into tensor memory) which is enormous surface area for a one-GEMM-at-a-time engine. Keep it on the radar only if bw24 adopts NVFP4 weights; otherwise it is far heavier than Marlin's hand-written m16n8k16 for the same single-stream job.
- **TMEM scale-factor layout machinery** (`blockscaled_layout.py:267-341`, `dense_blockscaled_gemm_sm120_b12x.py:159-166` manual-unroll workaround). Only meaningful with the block-scaled MMA above; the "hasAuxTensor address-space bug" workaround is dead weight for bw24.
- **SM103 expanded block-scale atom** (`blockscaled_layout.py:343-374`, `Sm103BlockScaledBasicChunk`) — different SM, DEAD.
- **`tcgen05` / `wgmma` (sm_90 / sm_100) paths** — none of the *selected* kernels above use them, but anything in the CUTLASS DSL that routes to UMMA/tcgen05 descriptors or warp-group MMA is DEAD on sm_120 (Ada has no wgmma, no tcgen05). bw24 must stay on warp-level `mma.sync`.
- **Multi-request server machinery / persistent multi-CTA cluster scheduling** (`dense_blockscaled_gemm_sm120_b12x.py:90-142` cluster setup). Cluster is forced to (1,1,1) on B12x anyway, and the persistent tile scheduler + NamedBarrier multi-CTA coordination is for throughput servers. Single-stream prefill = one kernel launch per GEMM; this scheduling is DEAD.
- **Marlin's large-M two-phase global reduce as a *server* feature** — the small-M atomicAdd path (idea #6 above) is useful; the large-M `C_tmp` staging across all SMs is tuned for big batched M and is low value for batch=1.
- **AMX** — not present in either tree; N/A (no x86 AMX kernels here). DEAD by absence.

## bw24 integration notes (single-stream sm_120 GGUF)

- **Adopt the Marlin core, skip the DSL.** The fastest credible path to closing the prefill gap is: GGUF Q4_K/Q4_0 -> Marlin repacked layout (offline, idea #4/#5) -> 4-stage cp.async.cg + ldmatrix pipeline (idea #3) feeding `mma.sync m16n8k16` (idea #1) with LOP3 dequant in the load (idea #2) and per-block scale on the FP32->FP16 epilogue (idea #5). Every instruction in that chain is sm_80+ and RUNS on sm_120 Ada.
- **FP32 reduce for long K.** Use Marlin's small-M atomicAdd + optional FP32 reduce (idea #6) so prefill over long context keeps accumulation accuracy without FP16 drift.
- **The CUTLASS sm_120 block-scaled path is a future NVFP4 option, not a GGUF path.** It RUNS on sm_120a but its entire value depends on NVFP4/MXFP4 block-scaled inputs + TMA/TMEM; for GGUF weights it is heavier surface area than Marlin for the identical single-stream job. Revisit only if bw24 ingests NVFP4 weights.
- **No wgmma, no tcgen05, no AMX.** Ada sm_120 has warp-level `mma.sync` only; anything routing to warp-group MMA (sm_90) or tcgen05 (sm_100) is DEAD.
